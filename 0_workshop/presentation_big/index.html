What is GraphQL

### A query language for your API ### A runtime for your server ###### http://graphql.org/code/#server-libraries C# / .NET, Clojure, Erlang, Go, Groovy, Java, JavaScript, PHP, Python, Scala, Ruby

It replaces your REST-endpoints

If you want.

Data model agnostic

Define your data as a graph

Define what exists, and how the data relates

Why GraphQL

  • Clean and strong typing of your API
  • Efficient - The client only get specifically what it asks for
  • Let your API evolve naturally
  • Normalize your data flow
### Clean and strong typing of your API ```javascript type Employee { id: String! name: String, employeeOfTheMonth: Boolean, } type Query { employees: [Employee]! employee(name: String!): Employee } ``` Everything has a type and you have to declare whether or not a type is nullable.
### The client only get specifically what it asks for ```javascript // query query { employee(name: "karl") { employeeOfTheMonth } } ``` ```javascript // response { "data": { "employee": { "employeeOfTheMonth": false } } } ```
### The client only get specifically what it asks for ```javascript // query query { employee(navn: "karl") { employeeOfTheMonth pay { yearly } } } ``` ```javascript // response { "data": { "employee": { "employeeOfTheMonth": false, "pay": { "yearly": 50 } } } } ```
### The client only get specifically what it asks for ```javascript query { employee(name: "karl") { name pay { yearly } // <--- Asks for pay } } ``` - The server has defined how pay is fetched from a employee-node. - Pay is only fetched when the client asks for it - Node-relations can be complex and circular, GraphQL will resolve until all values are "Scalars"
### Let your API evolve naturally ```javascript type Employee { id: String! name: String, employeeOfTheMonth: Boolean, } ``` ```javascript type Employee { id: String! name: String, employeeOfTheMonth: Boolean, pay: Int! // <--- New field } ```
### Let your API evolve naturally ```javascript type Employee { id: String! name: String, employeeOfTheMonth: Boolean @deprecated(reason: "Oof this should't have been available", pay: Int! } ```

Normalize your data flow

What is Apollo

### Simple way to consume a GraphQL API in a React application

Why Apollo

- Quick and easy to get started - Removes a lot of the responsibilty from Redux - Automatic state updates without complex reducers - Familiar if you have used HoCs (redux' `connect`, react-router's `withRouter`) - Also supports "function as children"-pattern
##### Connect your component, Apollo with handle fetching, loading state and any errors that occur ```javascript // EmployeeList: Normal component export default graphql(gql` query allEmployees { employees { id name employeeOfTheMonth } } `)(EmployeeList); ``` ```javascript // props: { data: { loading: false, employees: [ { name: "karl", employeeOfTheMonth: false }, ... ], error: null, refetch() { ... }, ... // a lot more } } ```
##### Automatic updating of state Give your data objects a globally unique ID and apollo will automatically update the state when it recognizes. ```javascript const client = new ApolloClient({ ..., dataIdFromObject: object => object.id, }); ```
##### Manual updating of state Are objects added or deleted you need to manually specify how Apollo should update the state - `update`: A function that allows you to read and write to the Apollo-cache. - `refetchQueries`: Be lazy and tell Apollo to get another query again.
